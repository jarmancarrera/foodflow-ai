⚡ The Root Cause

The problem isn't that people don't care — it's a timing + coordination failure. By the time a food bank is reached, the surplus is already cold or discarded. FoodFlow solves this by initiating the rescue 2 hours before closing — autonomously — so drivers arrive before food expires. No human coordination required at the institutional side.

The Solution — FoodFlow AI
Autonomous agent loop powered by Claude claude-sonnet-4-6 tool_use
🌿 FoodFlow AI
One-sentence explanation
FoodFlow connects to a restaurant's inventory system, detects surplus food 2 hours before closing, finds the nearest food bank with capacity, locates a volunteer driver, verifies legal compliance, and dispatches pickup — all automatically, in 4 seconds, with zero human intervention.

🤖 Claude's Role

Claude acts as the orchestration brain. It receives a surplus trigger and decides which tools to call, in what order, and what to do with their outputs. It reasons across all 6 steps using natural language and structured tool outputs — like a very fast, never-tired logistics coordinator.

⚡ Why 4 Seconds?

Each "tool call" is a simple Python function returning structured JSON. Claude doesn't wait for external APIs — it queries a local SQLite DB. Network latency is just the Anthropic API call (~1–2s). Total: DB queries (ms) + 2x Claude inference (~4s round trip).

📋 What Triggers It?

Three trigger sources: (1) Cron job 2h before closing time, (2) Manual "Rescue Now" button from dashboard, (3) Webhook from POS system when inventory crosses surplus threshold. All three route to the same agent loop.

💡 Why This Isn't Just a Script

A traditional script would require hard-coded logic: if capacity > X and distance < Y and item_type in approved_list… — brittle, unmaintainable. Claude uses natural language reasoning to handle edge cases: "The food bank at 2.1 miles is full, but their notes say they accept small packages after-hours — I'll route to the secondary option and flag for human review." This emergent judgment is impossible with pure rule-based systems.

The 6 Tools — Deep Dive
What each tool does, inputs/outputs, and why it matters
🌿 FoodFlow AI
📦 Tool 1: check_inventory
DB query
Input
location_id: str — which institution to check
Action
SELECT * FROM food_inventory WHERE location_id=? AND is_surplus=1 AND expiry_window_hours < 4
Output
List of surplus items with name, quantity_lbs, food_type, expiry_hours
Claude uses
Total lbs, food categories (hot/cold/dry) to query food bank capacity
🏦 Tool 2: check_foodbank_capacity
match + filter
Input
food_type: str, quantity_lbs: float, location_lat/lng: float
Action
Filter banks where capacity_lbs - current_load >= quantity. Sort by distance (haversine)
Output
Best matching food bank: name, address, distance_miles, available_capacity_lbs
Claude uses
Bank coordinates to pass to route calculator + volunteer finder
🙋 Tool 3: query_volunteers
geospatial
Input
supplier_lat/lng: float, max_distance_miles: float (default 10)
Action
Haversine distance from supplier to each available volunteer. Filter is_available=1. Sort ascending
Output
Nearest available driver: name, phone, distance_miles, vehicle_capacity_lbs
Claude uses
Driver phone for SMS dispatch; distance to cross-check route feasibility
🗺️ Tool 4: calculate_route
ETA engine
Input
volunteer_lat/lng, supplier_lat/lng, foodbank_lat/lng
Action
Haversine: volunteer→supplier + supplier→foodbank. Assume 20 mph avg urban speed
Output
total_distance_miles, estimated_minutes, pickup_eta, delivery_eta
Claude uses
ETA to verify food won't expire before arrival; include in SMS to driver
⚖️ Tool 5: verify_compliance
legal check
Input
food_type: str, preparation_method: str, storage_temp: str
Action
Rule-based check: Bill Emerson Good Samaritan Act (1996) protection eligibility
Output
is_compliant: bool, protection_level: str, liability_notes: str
Claude uses
Gate for dispatch — will NOT send driver if compliance fails. Logs reason
🚀 Tool 6: dispatch_pickup
final action
Input
volunteer_phone, supplier_address, foodbank_address, food_summary, eta_minutes
Action
Log pickup to DB (pickups table). Print mock SMS. Mark volunteer busy. Update inventory
Output
pickup_id: str, status: "dispatched", sms_sent: bool, timestamp
Claude uses
Confirms success in natural language summary returned to dashboard

How the Agent Loop Works
agent.py line-by-line explanation for non-engineers
🌿 FoodFlow AI
The Core Loop — Annotated
# agent.py — The brain of FoodFlow def run_agent(trigger: dict) -> dict: messages = [{ "role": "user", "content": f"Rescue surplus at {trigger['location']}" }] # Keep calling Claude until it says "done" while True: response = client.messages.create( model="claude-sonnet-4-6", max_tokens=4096, system=SYSTEM_PROMPT, tools=TOOL_SCHEMAS, # all 6 tools messages=messages ) # If Claude is done, return final message if response.stop_reason == "end_turn": return {"status": "complete", "summary": response.content[0].text} # Claude wants to call a tool for block in response.content: if block.type == "tool_use": # Execute the Python function result = TOOL_FUNCTIONS[block.name](**block.input) # Send result back to Claude messages.append({ "role": "user", "content": [{ "type": "tool_result", "tool_use_id": block.id, "content": json.dumps(result) }] })
Plain English Step-by-Step
STEP 1 — Initial Message
We tell Claude what to do
We send a natural language message: "There's surplus food at Cornell Statler Hall. Please coordinate a rescue." Claude reads this and decides its first action.
STEP 2 — Claude Responds with Tool Call
Claude asks for data
Instead of answering in text, Claude returns: "I want to call check_inventory with location_id='statler'". This is the tool_use block in the response.
STEP 3 — We Execute the Tool
Our code runs the function
We look up "check_inventory" in our TOOL_FUNCTIONS dict and call it with Claude's arguments. It queries SQLite and returns a JSON result with surplus items.
STEP 4 — We Send Result Back
We give Claude the data
We append the tool result to the conversation and call Claude again. Claude reads the inventory data and decides its next tool call.
STEPS 5–12 — Remaining Tools
Loop repeats 5 more times
Claude calls tools 2–6 in sequence, reading each result and building its understanding. At each step it can reason: "The food bank is full — I'll try the secondary option."
FINAL — stop_reason = "end_turn"
Claude signals completion
After dispatch_pickup succeeds, Claude returns a natural language summary: "Successfully dispatched Sarah Chen (0.8mi away) to pick up 45lbs from Statler Hall. ETA: 12 min. Delivery to Ithaca Food Bank confirmed."
